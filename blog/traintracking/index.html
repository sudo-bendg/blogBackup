Traintracking

Today I had my first encounter with the lovely folks at Homebrew Website Club in Edinburgh. There are exactly 3 things which I enjoyed about this meetup:

  1. A particularly good cortado
  2. Getting to meet some rather nice people
  3. Hearing about some genuinely interesting projects.

On this last point, I do emphasise the word genuinely. Projects which are not just interesting due to their outcomes, but also because of their motivations. Things which people make out of a sense of curiosity and enjoyment. This got me thinking, what could I do to make my website more than just a nice wrapper for my CV? More precisely, what do I actually want to share? All of these thoughts came to produce one final product: Traintracking.

What is it?

I enjoy trainspotting. It is potentially the least cool thing about me - but it is, alas, something about me. I would like to have a place to share the trains which I have seen, or rather to share the times at which I have seen trains. My thoughts are to have a list of objects with some details about a sighting. As simple as:

  • Number of train
  • Time of sighting
  • Location of sighting
    • Route travelled, if it was a train I was on

This data can be stored in my website’s existing sqlite database, and can be operated on through my existing Go backend. Indeed, the current functionality of this backend is to handle CRUD operations, and this is exactly what I need to implement Traintracking. I can worry about a nice presentation for this later on, for now, let’s put together a bit of a backlog to get a version 1.0.0 up and running.

Backlog 1

  1. Create a new train_sighting collection in the database
    • It should have schema
      • train_sighting_id: integer primary key auto increment
      • number: string not null
      • location1: string not null
      • location2: string
      • timestamp: date
    • I should be able to write rows into the table
    • I should be able to view existing rows in the table
  2. Create a POST /api/train_sighting endpoint
    • It should require a number and location1 value
      • It should return a 400 if these are not present
    • It should accept a location2 value
    • It should generate a timestamp value
    • It should create a row element if payload is valid
      • It should return a 201 if an element is created
  3. Create a GET /api/train_sighting endpoint
    • It should return a list of all sightings
  4. Create a traintracking page on bengodfrey.dev
    • It should make a GET request to /api/train_sighting
    • It should display the result of this request
    • It should implement pagination ensuring that at most 10 pages are shown at a time
